## 一、系统总体设计

### 1.1 项目背景与目标

#### 1.1.1 项目背景

随着高校学生数量的增加和自习需求的提升，传统的自习室管理方式（如现场排队、手工登记）已无法满足高效、公平的座位分配需求。学生经常面临以下问题：

- **占座现象严重**：学生提前用物品占座，导致座位资源浪费
- **排队时间过长**：高峰期学生需花费大量时间现场排队
- **座位利用率低**：缺乏实时信息，学生无法知悉座位空闲状态
- **管理效率低下**：管理员手工登记、统计工作量大且易出错

本项目旨在开发一套基于Web的自习室座位预约系统，实现在线预约、实时监控和智能管理，提升座位使用效率和用户体验。

#### 1.1.2 系统目标

1. **便捷预约**：提供在线自习室座位查询与预约服务，学生可随时随地完成预约
2. **资源优化**：实现座位资源的合理分配和高效利用，减少闲置浪费
3. **实时监控**：提供实时的座位状态展示，学生可直观了解自习室使用情况
4. **智能管理**：自动处理违约判定、权限管控、数据统计等管理任务
5. **数据驱动**：通过数据统计分析，为自习室运营决策提供依据

### 1.2 系统功能概述

系统分为**学生端**和**管理端**两大功能模块：

### 1.3 系统总体架构设计

#### 1.3.1 架构风格说明

本系统采用**B/S（Browser/Server）分层架构**，结合**前后端分离**的设计模式。整体分为四层：

- **表示层（Presentation Layer）**：基于Vue.js框架构建的SPA（单页面应用），运行在浏览器中，负责用户界面展示和交互
- **应用层（Application Layer）**：基于Spring Boot构建的RESTful API服务，处理HTTP请求，实现业务逻辑
- **数据访问层（Data Access Layer）**：使用MyBatis-Plus框架实现ORM映射，完成数据库CRUD操作
- **数据层（Data Layer）**：MySQL关系型数据库，存储系统所有业务数据

选择此架构的理由：

1. **分层架构**将系统职责清晰分离，每层只依赖其下层，降低了耦合度
2. **前后端分离**使前端和后端可独立开发、独立部署、独立测试，提高团队协作效率
3. **RESTful API**采用无状态设计，便于水平扩展和负载均衡
4. 各层可独立替换或升级（如更换前端框架或数据库），不影响其他层

### 1.4 技术选型说明

#### 1.4.1 技术栈总览

| 层次 | 技术 | 版本 | 说明 |
|------|------|------|------|
| **前端框架** | Vue.js | 3.4 | 渐进式JavaScript框架，组件化开发 |
| **UI组件库** | Element Plus | 2.6 | 成熟的Vue 3组件库，提供丰富UI组件 |
| **HTTP客户端** | Axios | 1.7 | Promise-based HTTP客户端 |
| **路由** | Vue Router | 4.3 | Vue.js官方路由管理器 |
| **状态管理** | Pinia | 2.1 | Vue 3官方推荐状态管理库 |
| **后端框架** | Spring Boot | 3.2 | 企业级Java开发框架 |
| **ORM框架** | MyBatis-Plus | 3.5 | 增强版MyBatis，简化CRUD操作 |
| **数据库** | MySQL | 8.0 | 关系型数据库管理系统 |
| **连接池** | HikariCP | 5.0 | Spring Boot默认高性能连接池 |
| **认证授权** | Spring Security + JWT | - | 基于JWT的无状态认证方案 |
| **API文档** | SpringDoc OpenAPI | 2.3 | 自动生成Swagger API文档 |
| **构建工具** | Maven | 3.9 | Java项目依赖管理与构建 |
| **前端构建** | Vite | 5.1 | 新一代前端构建工具 |
| **容器化** | Docker | 24+ | 应用容器化部署 |

#### 1.4.2 选型理由分析

**Vue.js 3 + Element Plus（前端）**
- Vue 3采用Composition API，代码组织更灵活，逻辑复用性强
- Element Plus提供丰富的企业级UI组件（表格、表单、弹窗、图表等），开发效率高
- Vite构建工具开发体验极佳，热更新速度远超Webpack
- 团队有Vue.js开发经验，学习成本低

**Spring Boot 3.2（后端）**
- 自动化配置减少了大量样板代码，开发效率高
- 成熟的生态系统（Security、AOP、Validation、Actuator等），满足企业级需求
- 内嵌Tomcat服务器，无需额外部署Web容器
- 课程教学常用框架，符合课程要求

**MyBatis-Plus（ORM）**
- 在MyBatis基础上提供通用CRUD接口，减少重复编码
- 支持Lambda表达式构建查询条件，类型安全且可读性强
- 与MySQL搭配性能优异，支持复杂SQL自定义

**MySQL 8.0（数据库）**
- 需求文档明确要求使用MySQL 8.0
- 开源免费，社区活跃，文档丰富
- 支持JSON数据类型和窗口函数等高级特性
- 适合中小型Web应用的数据存储需求

**JWT + Spring Security（认证）**
- JWT无状态，服务端无需存储会话信息，便于水平扩展
- Spring Security提供完整的认证授权框架，支持RBAC权限模型
- 两者结合可实现安全、高效的用户认证体系

## 二、详细设计

### 2.1 模块划分与模块功能设计

#### 2.1.1 模块总体划分

系统共划分为**6个核心模块**，各模块职责明确、边界清晰：

```
自习室座位预约系统
├── 1. 用户模块 (user-module)
├── 2. 自习室模块 (room-module)
├── 3. 座位模块 (seat-module)
├── 4. 预约模块 (reservation-module)
├── 5. 统计模块 (statistics-module)
└── 6. 公共模块 (common-module)
```

#### 2.1.2 模块职责说明

**1. 用户模块 (user-module)**

| 项目 | 内容 |
|------|------|
| **职责** | 负责用户注册、登录、认证、个人信息管理、违约记录查询 |
| **核心功能** | 用户注册、账号登录、JWT令牌签发与刷新、个人信息修改、密码修改、违约记录查看 |
| **输入** | 注册表单、登录凭证、个人信息编辑表单 |
| **输出** | JWT令牌、用户信息DTO、违约记录列表 |

**2. 自习室模块 (room-module)**

| 项目 | 内容 |
|------|------|
| **职责** | 负责自习室信息的增删改查、开放时间设置 |
| **核心功能** | 自习室列表查询、自习室详情查看、自习室新增/修改/删除（管理员）、开放/关闭状态切换 |
| **输入** | 自习室查询条件、自习室编辑表单 |
| **输出** | 自习室列表DTO、自习室详情（含座位概览） |

**3. 座位模块 (seat-module)**

| 项目 | 内容 |
|------|------|
| **职责** | 负责座位信息管理、座位状态实时更新 |
| **核心功能** | 座位布局展示、座位状态查询、座位新增/修改/删除（管理员）、座位类型管理、座位二维码生成 |
| **输入** | 自习室ID、座位筛选条件、座位编辑表单 |
| **输出** | 座位列表及状态、座位布局图数据 |

**4. 预约模块 (reservation-module)**

| 项目 | 内容 |
|------|------|
| **职责** | 负责预约的创建、取消、签到、签退以及违约判定 |
| **核心功能** | 座位预约、取消预约、扫码签到、签退释放、违约自动判定、预约规则校验、时段冲突检测 |
| **输入** | 预约请求（座位ID+时段）、取消请求、签到请求、签退请求 |
| **输出** | 预约记录、签到确认、违约记录、操作结果 |

**5. 统计模块 (statistics-module)**

| 项目 | 内容 |
|------|------|
| **职责** | 负责系统数据的统计分析和报表生成 |
| **核心功能** | 座位使用率统计、用户活跃度分析、时段热度分析、违约率统计、数据导出 |
| **输入** | 统计时间范围、统计维度选择 |
| **输出** | 统计图表数据、统计报表（Excel导出） |

**6. 公共模块 (common-module)**

| 项目 | 内容 |
|------|------|
| **职责** | 提供系统级通用功能和基础设施 |
| **核心功能** | JWT认证拦截、RBAC权限校验、统一异常处理、统一响应封装、操作日志记录（AOP）、参数校验、跨域配置 |
| **输入** | HTTP请求、业务异常、操作事件 |
| **输出** | 统一响应格式、日志记录、权限判断结果 |

#### 2.1.3 模块调用关系

```
                      ┌──────────────┐
                      │   用户模块    │
                      └──────┬───────┘
                             │ 认证通过后调用
         ┌───────────────────┼───────────────────┐
         ↓                   ↓                   ↓
  ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
  │  自习室模块   │   │   座位模块    │   │   预约模块    │
  └──────┬───────┘   └──────┬───────┘   └──────┬───────┘
         │                  │                  │
         │    查询自习室     │   查询座位状态    │   创建预约记录
         ↓                  ↓                  ↓
  ┌──────────────────────────────────────────────────────┐
  │                      数据库层                         │
  │   users  │  study_rooms  │  seats  │  reservations   │
  └──────────────────────────────────────────────────────┘
                             ↑
                    ┌──────────────┐
                    │   统计模块    │  ← 读取各表数据进行聚合分析
                    └──────────────┘

  所有模块 → 公共模块（认证、授权、日志、异常处理、响应封装）
```

模块间的调用遵循**单向依赖原则**：业务模块（用户、自习室、座位、预约）之间通过Service层进行调用，均依赖公共模块提供的横切功能。统计模块以只读方式访问其他模块的数据，不产生业务写入。

### 2.2 核心业务流程设计

#### 2.2.1 用户预约座位流程

```
开始
  │
  ▼
用户选择自习室
  │
  ▼
系统展示座位布局图（绿色=空闲，红色=已预约，黄色=使用中）
  │
  ▼
用户选择空闲座位和预约时段
  │
  ▼
系统校验预约资格
  ├── 今日预约次数 ≥ 3？ → 提示"今日预约次数已达上限" → 返回
  ├── 该时段已被预约？   → 提示"时段冲突" → 返回
  └── 校验通过
        │
        ▼
系统显示预约确认信息（自习室、座位号、时段）
  │
  ▼
用户确认
  │
  ▼
系统生成预约记录，座位状态更新为"已预约"
  │
  ▼
返回预约成功信息（含预约编号、签到二维码）
  │
  ▼
结束
```

#### 2.2.2 签到流程

```
开始
  │
  ▼
用户到达自习室，扫描座位二维码 / 点击"签到"按钮
  │
  ▼
系统获取当前时间和预约信息
  │
  ▼
签到时间校验
  ├── 早于开始时间15分钟以上 → 提示"尚未到签到时间" → 返回
  ├── 晚于开始时间15分钟以上 → 记为违约 → 座位释放
  └── 在允许签到窗口内
        │
        ▼
系统记录签到时间，座位状态更新为"使用中"
  │
  ▼
返回签到成功信息
  │
  ▼
结束
```

#### 2.2.3 违约处理流程

```
定时任务（每5分钟执行）
  │
  ▼
查询所有"已预约"状态且超过签到窗口的记录
  │
  ▼
将这些记录标记为"违约"
  │
  ▼
释放对应座位（状态变为"空闲"）
  │
  ▼
检查用户连续违约次数
  ├── 连续违约 ≥ 3次 → 暂停预约权限7天 → 推送处罚通知
  └── 否则 → 记录违约次数
        │
        ▼
结束
```

#### 2.2.4 取消预约流程

```
开始
  │
  ▼
用户进入"我的预约"页面
  │
  ▼
选择要取消的预约
  │
  ▼
取消时间校验
  ├── 距开始时间 < 30分钟 → 提示"距离开始不足30分钟，无法取消" → 返回
  └── 距开始时间 ≥ 30分钟
        │
        ▼
弹出确认对话框
  │
  ▼
用户确认取消
  │
  ▼
系统释放座位（状态变为"空闲"），预约状态变为"已取消"
  │
  ▼
返回取消成功信息
  │
  ▼
结束
```

## 五、其他内容

### 5.1 安全性设计

#### 5.1.1 身份认证

系统采用**JWT（JSON Web Token）+ Spring Security**实现用户身份认证：

```
用户登录 → 验证用户名密码 → 签发JWT令牌 → 后续请求携带Token → 过滤器验证Token
```

- JWT令牌包含：userId、username、role、过期时间
- Access Token有效期：2小时
- Refresh Token有效期：7天，用于无感刷新Access Token
- 令牌存储在浏览器localStorage中，每次请求通过Authorization Header携带

#### 5.1.2 权限控制（RBAC）

系统采用基于角色的访问控制（Role-Based Access Control），区分两种角色：

| 角色 | 权限范围 |
|------|---------|
| **student** | 查看自习室、查看座位、预约座位、取消预约、签到签退、查看个人记录 |
| **admin** | 全部权限：用户管理、自习室管理、座位管理、预约管理、数据统计、系统设置 |

实现方式：
- 使用Spring Security的`@PreAuthorize`注解进行方法级权限控制
- 管理员接口使用`@PreAuthorize("hasRole('admin')")`进行拦截
- 用户只能操作自己的预约记录（数据权限隔离）

#### 5.1.3 数据安全

| 安全措施 | 实现方式 |
|---------|---------|
| **密码加密** | 使用BCrypt算法进行单向加密存储，不可逆 |
| **HTTPS传输** | 生产环境强制HTTPS，配置SSL证书 |
| **SQL注入防护** | 使用MyBatis-Plus参数化查询，杜绝SQL拼接 |
| **XSS防护** | 前端输出编码，后端配置CSP（Content Security Policy）头 |
| **CSRF防护** | JWT令牌天然防CSRF（不依赖Cookie） |
| **请求频率限制** | 使用Spring拦截器实现IP级别限流（如登录接口5次/分钟） |
| **会话管理** | JWT无状态设计，Token过期自动失效 |

#### 5.1.4 操作审计

- 使用AOP（面向切面编程）记录所有关键操作日志
- 记录内容：操作人、操作类型、操作时间、IP地址、操作详情
- 日志存储到独立的操作日志表，定期归档

### 5.2 性能优化设计

#### 5.2.1 数据库优化

| 优化措施 | 说明 |
|---------|------|
| **合理索引** | 在seat_id、user_id、reserve_date等高频查询字段建立索引 |
| **连接池** | 使用HikariCP连接池，最大连接数20，最小空闲连接数5 |
| **分页查询** | 所有列表查询接口强制分页，默认每页20条，最大100条 |
| **慢查询监控** | 开启MySQL慢查询日志，阈值设为1秒，定期优化 |

#### 5.2.2 应用层优化

| 优化措施 | 说明 |
|---------|------|
| **缓存策略** | 自习室列表、座位布局等变化频率低的数据使用Redis缓存，TTL 5分钟 |
| **异步处理** | 统计报表生成、日志记录等非实时操作使用`@Async`异步执行 |
| **定时任务优化** | 违约判定定时任务每5分钟执行一次，使用分布式锁避免重复执行 |
| **前端优化** | Vue路由懒加载，组件按需导入，图片压缩，CDN加速静态资源 |

#### 5.2.3 并发控制

预约场景存在并发竞争（同一座位同一时段可能被多人同时预约），采用以下方案：

1. **数据库乐观锁**：在seats表增加version字段，更新时校验版本号
2. **唯一索引约束**：reservations表建立(seat_id, reserve_date, time_slot)联合唯一索引
3. **业务兜底**：创建预约时在事务中先查询后插入，利用数据库行锁保证一致性

### 5.3 可扩展性设计

#### 5.3.1 水平扩展

- **无状态服务**：后端API服务不存储会话信息，可水平扩展至多个实例
- **Nginx负载均衡**：使用Nginx配置upstream，分发请求至多个Spring Boot实例
- **数据库读写分离**：预留MyBatis读写分离配置接口，后续可接入MySQL主从复制

#### 5.3.2 模块化设计

- **代码层面**：采用Maven多模块结构（common、service、web），模块间低耦合
- **接口层面**：所有模块通过Service接口通信，实现类可独立替换
- **配置层面**：业务规则（预约次数上限、签到窗口、违约阈值等）抽取为配置项，支持动态调整

#### 5.3.3 预留扩展点

| 扩展点 | 预留方式 |
|--------|---------|
| **微信小程序接入** | API层与前端解耦，后续可在不修改后端的情况下接入微信小程序 |
| **消息通知** | 预留通知接口（短信/邮件/微信推送），当前先通过系统内消息展示 |
| **支付系统** | 若未来需要付费预约，预留支付接口集成点 |
| **第三方登录** | 认证模块设计预留OAuth2扩展接口 |

### 5.4 异常处理机制

#### 5.4.1 异常分类

系统定义三层异常体系：

```
RuntimeException
├── BusinessException（业务异常）
│   ├── 重复预约异常
│   ├── 预约超限异常
│   ├── 签到超时异常
│   ├── 取消超时异常
│   └── 权限不足异常
├── ValidationException（参数校验异常）
└── SystemException（系统异常）
    ├── 数据库异常
    ├── 网络异常
    └── 服务不可用异常
```

#### 5.4.2 统一异常处理

使用Spring的`@RestControllerAdvice`实现全局异常拦截：

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(BusinessException.class)
    public Result handleBusinessException(BusinessException e) {
        // 返回业务错误信息，HTTP 409
        return Result.error(409, e.getMessage());
    }

    @ExceptionHandler(ValidationException.class)
    public Result handleValidationException(ValidationException e) {
        // 返回参数校验错误，HTTP 400
        return Result.error(400, "参数校验失败: " + e.getMessage());
    }

    @ExceptionHandler(Exception.class)
    public Result handleSystemException(Exception e) {
        // 记录详细日志，返回统一的系统错误提示，HTTP 500
        log.error("系统异常", e);
        return Result.error(500, "系统繁忙，请稍后重试");
    }
}
```

#### 5.4.3 前端异常处理

- Axios响应拦截器统一处理HTTP错误（401跳转登录、403提示无权限、500通用提示）
- 关键操作使用try-catch捕获异常并展示友好提示
- 网络超时设置10秒，超时后提供重试按钮

### 5.5 日志与监控方案

#### 5.5.1 日志方案

采用**SLF4J + Logback**日志框架，按照以下级别和策略记录：

| 日志级别 | 记录内容 | 示例 |
|---------|---------|------|
| **ERROR** | 系统异常、数据库连接失败、第三方服务调用失败 | 堆栈信息、请求参数 |
| **WARN** | 业务警告（违约、权限不足）、降级操作 | 用户ID、警告类型 |
| **INFO** | 关键业务操作（注册、预约、签到）、定时任务执行 | 操作人、操作类型、结果 |
| **DEBUG** | 开发调试信息、SQL语句（仅开发环境） | 方法入参、返回值 |

**日志输出：**
- 控制台输出（开发环境）
- 文件滚动存储（生产环境）：`logs/seat-reservation.%d{yyyy-MM-dd}.log`
- 日志保留30天，单文件最大100MB
- 使用MDC（Mapped Diagnostic Context）记录请求TraceId，便于链路追踪

#### 5.5.2 系统监控

使用**Spring Boot Actuator**暴露健康检查和指标端点：

| 端点 | 说明 |
|------|------|
| `/actuator/health` | 健康检查（数据库连接、磁盘空间） |
| `/actuator/metrics` | 应用指标（请求数、响应时间、JVM内存） |
| `/actuator/info` | 应用信息（版本号、构建时间） |

**关键监控指标：**
- API QPS（每秒请求数）和响应时间P99
- JVM堆内存使用率（告警阈值：>85%）
- 数据库连接池使用率（告警阈值：>80%）
- 活跃用户数、预约量（业务监控）
- 磁盘使用率（告警阈值：>90%）
